# PPT 自动生成 Web 系统设计文档

## 1. 文档目标

本文档用于定义一个面向特定业务场景的 PPT 自动生成 Web 系统的完整设计方案，覆盖以下内容：

- 业务目标与产品边界
- 核心功能需求与非功能需求
- 用户角色与用户交互流程
- 页面规划、布局、功能和交互细节
- 多 Agent + ReAct 技术架构
- 数据模型、接口设计、状态流转
- 模板体系、图片智能处理、自绘图形策略
- 开发实施建议、阶段划分、风险与降级策略

本文档目标不是停留在概念层，而是提供一份可以直接指导项目落地、任务拆分和后续研发实现的方案。

---

## 2. 产品定位

### 2.1 产品定义

本系统是一个通过 Web B/S 方式使用的 PPT 自动生成工具，主要服务于特定行业或有限用户群体。它不负责开放式地替用户完成完整知识采集，而是围绕“用户已具备原始材料”的前提，完成以下工作：

- 接收用户提供的原始内容
- 将内容整理为适合 PPT 展示的结构化大纲
- 自动决定每页主题、页间逻辑关系与页数分配
- 结合模板和视觉策略生成页面级 SVG 幻灯片
- 支持用户通过自然语言聊天持续修改已有页面或新增后续页面
- 最终导出为 PPT 文件

### 2.2 产品边界

本系统负责：

- 解析 TXT、Markdown、PDF、URL 等输入内容
- 基于页数目标和业务语境生成 PPT 大纲
- 根据模板系统完成版式匹配
- 判断是否需要图片、图表或图形
- 在适合时自绘图表、结构图、信息图
- 在无法自绘时进行网络搜图和图片理解筛选
- 生成可编辑的页面结果并导出 PPT

本系统不负责：

- 作为通用搜索引擎进行大范围开放研究
- 替代专业设计师进行像素级人工排版
- 实现复杂 PPT 原生动画、触发器和母版体系
- 完整替代 Office 级别的所见即所得编辑器

### 2.3 目标价值

- 降低从原始资料到可展示 PPT 的时间成本
- 让非设计人员也能快速得到结构较完整、风格一致的初稿
- 通过多 Agent 协作提升质量，而不是单次生成的一次性结果
- 通过聊天式迭代降低二次修改门槛

## 3. 用户与使用场景

### 3.1 目标用户

- 咨询顾问、方案经理、售前、行业分析人员
- 需要频繁产出固定风格汇报材料的团队成员
- 已有企业内部模板/视觉规范，但缺少自动化生产能力的组织

### 3.2 典型使用场景

场景一：基于报告和 PDF 生成汇报材料

- 用户上传 PDF 与一段说明文字
- 设定目标为 10 页
- 系统分析文档并规划 10 页结构
- 用户确认大纲后生成初版 PPT
- 用户通过聊天修改若干页面并导出

场景二：基于网页链接与文字要求生成方案型 PPT

- 用户输入多个 URL 与目标页数
- 系统抽取网页正文并生成方案型目录结构
- 对适合的页面自动插入示意图或信息图

场景三：基于长文本说明快速形成内部汇报

- 用户粘贴较长的会议纪要或分析结果
- 系统提炼成章节、分论点、结论页和行动建议页
- 用户继续补充“再加一页风险分析”之类的后续页面需求

## 4. 总体产品设计原则

### 4.1 质量优先于速度

本系统不采用“单轮对话直接输出 PPT”的方式，而采用多 Agent + ReAct 的逐步生成模式，以换取更高的大纲质量、版式匹配度和视觉准确性。

### 4.2 用户控制优先

- 大纲阶段设置人工确认节点
- 生成后保留可持续编辑能力
- 聊天输入支持区分“修改现有内容”和“新增后续页面”

### 4.3 模板驱动而非完全自由生成

由于目标用户群体较垂直，系统应围绕用户自建模板库工作，追求风格稳定和结果可控，而不是完全自由风格生成。

### 4.4 视觉能力分层处理

视觉内容按能力分层：

- 能自绘的图表、结构图、简单信息图，优先自绘
- 不适合自绘的真实图片、场景照片，通过搜索和视觉理解筛选

### 4.5 Web 端完成主要交互

系统核心体验均在 Web 中完成，包括：

- 项目管理
- 内容输入
- 生成过程观察
- 大纲确认
- 幻灯片预览与聊天编辑
- 导出 PPT

## 5. 核心功能需求

### 5.1 项目管理

系统需要具备完整的 PPT 项目管理能力。

功能要求：

- 创建新的 PPT 项目
- 查看历史项目列表
- 打开项目继续编辑
- 重命名项目
- 复制已有项目
- 删除项目
- 保存每个项目独立的页面数据、聊天记录和生成历史

业务规则：

- 每个项目是一个独立 session 的容器
- 一个项目对应一份 Presentation 数据与一条主聊天线程
- 项目创建后，进入该项目的内容输入与生成流程

### 5.2 内容输入与资料接收

系统支持以下输入方式：

- 粘贴大段文本
- 上传 TXT 文件
- 上传 Markdown 文件
- 上传 PDF 文件
- 输入网页链接 URL

系统需要：

- 提取文本正文
- 清洗噪声内容
- 允许用户补充生成意图说明
- 允许设置期望页数
- 允许补充“目标风格/目标用途/受众对象”等约束描述

### 5.3 大纲生成

系统基于内容与目标页数生成结构化大纲。

输出内容至少包括：

- 总标题建议
- 页面数量建议
- 每页主题
- 每页核心要点摘要
- 每页可能适用的模板类型
- 每页是否建议使用图像或图形

大纲生成不是一次定稿，而是多 Agent 规划后的中间结果。

### 5.4 大纲确认

大纲生成后必须暂停，等待用户确认。

用户可进行的操作：

- 调整页数
- 修改某一页主题
- 删除某一页
- 新增某一页
- 调整顺序
- 直接输入新的结构性要求

确认后系统才继续进入页面渲染阶段。

### 5.5 页面生成与模板应用

系统根据大纲、模板库、版式规则生成每一页的 SVG 幻灯片。

要求：

- 每页都从模板库中选择最合适模板
- 文本需适配对应 slot 的尺寸与字数限制
- 对溢出内容进行重写或压缩，而不是简单截断
- 保证整份 PPT 在视觉风格上统一

### 5.6 图片与图形处理

系统应具备“判断是否需要视觉元素”的能力。

若需要视觉元素，分为两类：

- 自绘图形
- 搜索真实图片

自绘范围包括：

- 柱状图、饼图、折线图
- 流程图、架构图、节点箭头图
- 图标与几何图形组合
- 时间轴、对比结构、简单信息图

当无法自绘或自绘不合适时，系统需：

- 自动形成图片搜索关键词
- 调用配置好的图片搜索服务
- 获取若干候选图片
- 使用视觉模型评分最匹配图片
- 将最佳图片插入相应 slot

### 5.7 聊天式二次编辑

生成后，用户在编辑器内通过右侧聊天面板发起修改。

聊天需支持两类意图：

- EDIT：修改当前已有页面
- ADD：新增后续页面

示例：

- “把第 3 页标题改成更正式一些” -> EDIT
- “在这页后面再增加一页风险分析” -> ADD

系统要求：

- 自动识别聊天意图
- 在 UI 中以标签展示意图结果
- 用户可以手动纠正意图标签
- EDIT 默认作用于当前选中页
- 若未选中页面，则尝试从用户语句中识别页码或页名
- ADD 默认插入到当前页之后，若无选中页则追加到结尾

### 5.8 导出 PPT

导出时每一页 PPT 对应一张生成好的 SVG 页面。

推荐方案：

- 在浏览器端将 SVG 渲染为 Canvas
- 再转为 PNG
- 使用 PPTX 生成库写入每页全屏图片

要求：

- 导出结果可被 Microsoft PowerPoint 正常打开
- 页面比例与 Web 预览一致
- 导出过程在用户可接受范围内完成

## 6. 非功能需求

### 6.1 可维护性

- 架构要支持后续增加新的 Agent、模板类型和图片搜索 Provider
- 前后端需要清晰分层
- Prompt、模板 schema、Agent 配置应尽量模块化

### 6.2 可解释性

生成过程要能让用户感知系统正在做什么。

要求：

- 提供 Agent 进度流
- 允许查看摘要级 ReAct 执行信息
- 错误时能定位是哪个阶段失败

### 6.3 性能

- 页面切换与缩略图浏览要流畅
- 大纲生成和渲染过程中要持续反馈进度
- 图片搜索和视觉评分阶段需要做并发控制，避免接口风暴

### 6.4 一致性

- 同一项目内字体、颜色、版式风格一致
- 同一类页面尽可能使用一组风格近似模板

### 6.5 安全性

- 对用户上传内容做基础校验
- 对 URL 抓取进行超时与域名控制
- 对 SVG 注入内容进行 sanitize
- 对第三方图片 URL 做安全代理或缓存策略

## 7. 技术方案总览

### 7.1 技术选型

- 前后端框架：Next.js 14 + TypeScript
- UI：React + Tailwind CSS
- 数据库：SQLite + Prisma
- LLM 调用：OpenAI SDK，兼容 OpenAI 风格接口，可接 DeepSeek/Qwen 等
- PDF 解析：pdfjs-dist
- URL 抽取：服务端抓取 + cheerio
- SVG 渲染：React SVG 组件 + 模板 slot 注入
- 导出 PPT：canvg + pptxgenjs

### 7.2 选型理由

选择 Next.js 的原因：

- 适合同时承载页面与 API
- 降低前后端拆分管理成本
- 支持流式响应，适合 Agent 进度回传

选择 SQLite + Prisma 的原因：

- 适合第一阶段落地
- 无需复杂运维
- 对单机或小范围使用足够稳定

选择自定义 Agent 编排而不是引入复杂框架的原因：

- 当前流程高度定制化
- 需要明确掌控各 Agent 输入输出与反馈回路
- 第一阶段优先可控与可调试

## 8. 多 Agent + ReAct 架构设计

### 8.1 为什么采用多 Agent

如果让单个 LLM 在一轮中直接生成整份 PPT，会出现以下问题：

- 大纲和页面内容容易失衡
- 版式适配不可控
- 视觉素材决策质量不稳定
- 出错时难以定位和纠正

多 Agent 方案的目标是将复杂问题拆解为多个职责明确的任务单元，每个 Agent 针对单一问题做更高质量判断。

### 8.2 Agent 列表

1. OrchestratorAgent
2. ContentExtractionAgent
3. OutlinePlannerAgent
4. ContentWriterAgent
5. LayoutSelectorAgent
6. VisualDecisionAgent
7. GraphicGeneratorAgent
8. ImageSearchAgent
9. QualityReviewAgent

### 8.3 各 Agent 职责定义

#### 8.3.1 OrchestratorAgent

职责：

- 接收项目目标
- 按阶段调度其他 Agent
- 维护全局状态
- 处理重试与回退
- 控制 revision 次数上限
- 通过 SSE 向前端输出进度

输入：

- 用户输入资料
- 项目配置
- 当前任务状态

输出：

- Agent 调度动作
- 阶段性结果
- 最终 deck 数据

#### 8.3.2 ContentExtractionAgent

职责：

- 解析 PDF、URL、文本
- 清洗和切分内容
- 判断内容是否不完整
- 必要时触发辅助搜索行为

输出：

- ExtractedContent

#### 8.3.3 OutlinePlannerAgent

职责：

- 生成页面级大纲
- 检查结构平衡性
- 检查页数是否匹配
- 检查逻辑顺序
- 自我修正后输出更优版本

这是整个流程中的关键 Agent。

#### 8.3.4 ContentWriterAgent

职责：

- 为每一页生成标题、副标题、要点、说明文字
- 控制内容长度
- 满足模板 slot 的字符上限和版式约束

#### 8.3.5 LayoutSelectorAgent

职责：

- 为每一页选择最匹配模板
- 检测文字溢出与排版冲突
- 当内容不适合模板时向上游发起 revision 请求

#### 8.3.6 VisualDecisionAgent

职责：

- 判断每页是否需要视觉元素
- 判断适合使用自绘还是图片搜索
- 产出视觉 brief

#### 8.3.7 GraphicGeneratorAgent

职责：

- 生成适配 slot 尺寸的 SVG 图表或图形
- 对尺寸和可读性进行反复修正

#### 8.3.8 ImageSearchAgent

职责：

- 生成关键词
- 调用搜索服务
- 拉取候选图
- 借助视觉模型评分并选优

#### 8.3.9 QualityReviewAgent

职责：

- 对整套结果做最终审查
- 检查页面一致性、图文匹配、缺漏页、风格冲突
- 触发定向返工

### 8.4 ReAct 执行模式

每个 Agent 采用如下模式执行：

1. Thought：当前阶段应如何判断
2. Action：调用何种工具或接口
3. Observation：拿到结果后如何理解
4. Loop：若结果不达标则继续下一轮

工程要求：

- Thought 不需要对用户完全透明，但要保留摘要日志
- 每个 Agent 设置最大 Thought 循环次数
- 每个 feedback loop 设置最大 revision 次数，建议为 2

### 8.5 Agent 之间的回压机制

示例：

- LayoutSelectorAgent 发现内容超出模板容纳能力，要求 ContentWriterAgent 精简文字
- QualityReviewAgent 发现第 5 页与全局结构不一致，要求 OutlinePlannerAgent 或 LayoutSelectorAgent 局部重做

原则：

- 允许回压
- 不允许无限循环
- 超过阈值时采取降级策略

### 8.6 人工确认节点

建议至少保留两个关键节点：

- 节点一：大纲规划完成后，用户确认再继续
- 节点二：全量页面生成完成后，用户进入聊天编辑阶段

第一版可以只强制保留节点一，节点二作为正常编辑入口。

## 9. 模板系统设计

### 9.1 模板来源

模板主要来源于用户自建参考库，而不是完全由系统默认生成。

用户可提供：

- 参考截图
- 口述布局描述
- 既有 PPT 元素库

### 9.2 模板结构

建议每个模板由两个文件组成：

- 一个 SVG 骨架文件
- 一个 JSON schema 文件

SVG 骨架负责静态视觉框架。

Schema 负责定义：

- slot 名称
- slot 类型
- slot 的边界框
- 最大字数或布局约束
- 对应可接受的内容类型

示例 slot：

- slot-title
- slot-subtitle
- slot-body
- slot-bullets
- slot-image
- slot-chart

### 9.3 模板导入方式

支持两种导入路径：

- 路径一：上传截图，由视觉模型解析布局并生成 SVG 骨架与 schema 初稿
- 路径二：用户通过描述说明布局结构，系统辅助生成模板初稿

第一版可先提供“截图导入 + 手工校验”的路径。

### 9.4 模板分类

建议模板按用途进行标签分类：

- 封面页
- 目录页
- 章节过渡页
- 结论页
- 纯文本分析页
- 图文混排页
- 双栏对比页
- 数据图表页
- 流程页
- 风险与建议页

## 10. 图片与图形智能处理设计

### 10.1 视觉判断流程

每页在生成时执行一次视觉决策：

1. 是否需要视觉元素
2. 若需要，是图表/结构图/信息图还是外部图片
3. 若为图表或结构图，是否能在当前模板 slot 中自绘
4. 若不能自绘，则进入图片搜索流程

### 10.2 自绘策略

适合自绘的情况：

- 数据有明显结构
- 需要表达逻辑流程
- 需要表达对比关系或时间线
- 只需要简洁示意图，不需要真实场景照片

输出要求：

- 生成符合模板 bbox 的 SVG
- 控制元素数量，避免图形过密
- 保持与整体模板风格一致

### 10.3 搜图策略

适合搜图的情况：

- 页面需要现实场景照片
- 页面需要人物、办公场景、行业场景图
- 页面需要视觉氛围支撑而不是结构表达

流程：

1. 生成关键词
2. 选择配置好的搜索 Provider
3. 获取候选图片
4. 视觉模型结合页面上下文进行评分
5. 选择分数最高的结果

### 10.4 降级策略

如果出现以下情况：

- 搜图失败
- 图片版权状态不明确
- 候选图片评分过低

则降级为：

- 使用抽象形状或默认图标占位
- 或在该页取消图片，保留纯文本结构

## 12. 页面规划与页面级设计

本系统建议至少规划 5 个主要页面。

### 12.1 页面一：项目列表页

路由建议：

- /

页面目标：

- 展示用户已有 PPT 项目
- 支持新建与管理项目

核心布局：

- 顶部：系统名称、搜索框、创建项目按钮
- 中间：项目卡片网格或列表

## 13. 关键用户交互流程

### 13.1 新建 PPT 项目流程

1. 用户进入项目列表页
2. 点击“新建项目”
3. 输入项目名称
4. 系统创建项目记录并进入 setup 页面

### 13.2 初次生成流程

1. 用户在 setup 页面上传资料或输入文本
2. 输入目标页数与附加要求
3. 点击开始生成
4. 系统进入 generating 页面
5. 多 Agent 开始执行
6. 大纲生成完成后暂停
7. 用户确认或修改大纲
8. 系统继续生成页面与视觉内容
9. 生成完成后跳转到编辑页

### 13.3 编辑现有页面流程

1. 用户在左栏选择第 N 页
2. 在右栏输入修改要求
3. 系统识别为 EDIT
4. 修改对应页面的数据与 SVG
5. 左栏缩略图与中栏预览同步更新

### 13.4 新增后续页面流程

1. 用户选中某页
2. 在右栏输入“在这页后面增加一页...”
3. 系统识别为 ADD
4. 生成新的大纲片段和页面内容
5. 将新页插入当前页之后
6. 左栏和中栏更新，新页可继续编辑

### 13.5 导出流程

1. 用户点击右栏顶部“导出 PPT”
2. 系统逐页将 SVG 转为图片
3. 组装 PPTX 文件
4. 触发浏览器下载

## 19. 风险点与应对策略

### 19.1 风险：LLM 输出不稳定

应对：

- 使用结构化 JSON 输出
- 每个 Agent 输出后做 schema 校验
- 失败时局部重试

### 19.2 风险：版式溢出

应对：

- 模板 slot 提前定义 max constraints
- LayoutSelectorAgent 与 ContentWriterAgent 协同修正

### 19.3 风险：搜图质量不稳定

应对：

- 候选图评分
- 引入阈值
- 失败时降级为纯文本或抽象图形

### 19.4 风险：多 Agent 令成本过高

应对：

- 控制每个 Agent 的最大迭代次数
- 对稳定步骤采用缓存
- 对部分页面并行处理但限制并发度

### 19.5 风险：前端 SVG 渲染复杂

应对：

- 第一版控制模板复杂度
- 优先实现稳定 slot 注入模型
- 复杂动效与高级编辑功能后置

## 20. MVP 建议范围

如果以尽快落地为目标，建议 MVP 包含以下能力：

- 5 个页面
- 项目 CRUD
- 文本/PDF/URL 输入
- 多 Agent 大纲生成
- 大纲确认
- 基于模板库的 SVG 页面生成
- 三栏编辑页
- EDIT/ADD 聊天模式
- SVG 导出为 PPTX

MVP 暂不强求：

- 高级模板编辑器
- 复杂拖拽交互
- 完整的登录权限体系
- 非常复杂的图形绘制语义

## 21. 最终建议

从落地角度看，这个系统最重要的不是“让模型一次生成一份 PPT”，而是建立一条质量可控、可回退、可解释、可持续编辑的生产链路。

因此，建议实施重点按以下优先级推进：

1. 先把项目管理、内容输入、大纲确认、三栏编辑和导出闭环做通
2. 再强化多 Agent 的回压能力和模板适配质量
3. 最后扩展模板导入、自绘图形和搜图评分能力

如果按这个顺序推进，项目能够较快进入“可内部试用”的状态，同时保留后续向高质量版本演进的空间。
